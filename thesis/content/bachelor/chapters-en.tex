\chapter{Introduction}\label{ch:b-introduction}
\section{A concrete problem}
At 02:14, a camera in an industrial room detects movement near a restricted cabinet. The network link is temporarily unavailable, so a cloud-only service cannot return a decision within the 50~ms response budget. A local model can respond in time, but only if memory use remains below 300~MB and the false alarm rate stays within the operational limit. This running example turns a broad topic into a measurable research problem.

Edge computing offers a useful foundation for local decision making, while its security boundary and resource constraints remain important design concerns \cite{shi2016edge,roman2018mobile}. The thesis therefore studies the combined performance, resource, and security requirement rather than optimising one metric in isolation.

\begin{researchquestion}
Can a compact edge device deliver an accurate decision within 50~ms and 300~MB while keeping inference and security policy as independently testable components?
\end{researchquestion}

\section{Central claim and contributions}
\begin{resultbox}[Central thesis claim]
A four-stage edge pipeline can satisfy the illustrative response and memory targets while preserving useful predictive performance, provided that each stage is measured separately and its assumptions are recorded.
\end{resultbox}

The thesis makes four refutable contributions:
\begin{enumerate}
  \item it defines the scenario, assumptions, and success criteria in Chapter~\ref{ch:b-problem};
  \item it implements a modular architecture with explicit interfaces in Chapter~\ref{ch:b-design};
  \item it evaluates accuracy, latency, memory, uncertainty, and failure cases against a baseline in Chapter~\ref{ch:b-evaluation};
  \item it positions the result against edge-computing and security literature and states the limits of the comparison in Chapter~\ref{ch:b-related}.
\end{enumerate}

\chapter{Problem and Running Example}\label{ch:b-problem}
\section{Operational scenario}
The prototype receives an image or sensor sample, computes a local score, applies a security policy, and displays an alert. The design must still produce a safe outcome when the cloud link is unavailable. The running example exposes three competing goals: a low response time, bounded resource use, and a defensible policy boundary.

\section{Assumptions and success criteria}
A claim is meaningful only when its scope is visible. This example assumes a fixed edge device, a versioned test dataset, a warmed-up model, and an isolated local network. It does not claim performance on every device or under adversarial physical manipulation.

\section{Interfaces and implementation}
The inference component receives a validated sample and returns both a decision and a score. The policy component can therefore change thresholds or access rules without retraining the model. Record hardware, operating-system version, library versions, model checksum, configuration parameters, and the command used to start each component.

\begin{verbatim}
def classify(sample, model, threshold=0.75):
    score = model.predict(sample)
    return int(score >= threshold), float(score)
\end{verbatim}

\begin{resultbox}[Design evidence]
A component-replacement test should show that the policy can be changed while the inference output remains identical. This test supports the modularity claim more directly than an architecture diagram alone.
\end{resultbox}

\section{Observation, interpretation, and uncertainty}
Across five illustrative runs, the optimised configuration meets all three numerical targets. A real thesis should also report dispersion or confidence intervals, not only point estimates. Explain whether the measured difference is large enough to matter in the operational scenario.

\begin{keypoint}[Example interpretation]
The latency and memory measurements support the claim on the tested hardware. They do not establish performance on other processors, thermal conditions, or workloads. Predictive accuracy supports the quality target only for the stated test distribution.
\end{keypoint}

\section{Failure case}
During a 20-minute run, an uncooled device may throttle and miss the latency target. Report the failure and restrict the claim to suitable operating conditions.

\chapter{Related Work and Positioning}\label{ch:b-related}

\chapter{Conclusions}
The example research question can be answered conditionally: on the tested device and workload, the modular prototype reaches the illustrative latency, memory, and accuracy targets. The component-replacement test supports the separation claim, while the thermal failure case limits the responsiveness claim to suitable operating conditions.

\section{Limitations and future work}
The example does not cover adversarial inputs, long-term field deployment, energy consumption, accessibility of the alert interface, or generalisation to other hardware. Concrete next steps are a broader dataset, an explicit threat model, energy measurement, deployment on alternative devices, and a user study of the alert workflow.
